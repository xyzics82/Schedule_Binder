# Google 캘린더 연동 지시서

> 이 문서는 Life Binder Web 앱에 Google 캘린더 연동을 추가하기 위한 작업 지시서입니다.
> **§1, §3-2 등 주요 결정은 이미 사용자가 선택을 마쳤고 ✅로 표시되어 있습니다.**
> 아직 비어있는 **`[ 입력: ... ]`** 항목만 채워서 다시 보내주시면 작업을 시작합니다.

---

## 사용 시나리오 (이 문서의 전제)

- **주 사용 도구**: 데스크톱에서 **Life Binder(LB) 앱**으로 일정 관리
- **보조 사용 도구**: 이동 중 **모바일에서는 Google 캘린더(GCal)**가 편함
- **연동 목적**: 데스크톱에서 만든 LB 계획을 모바일 GCal에서 보고, 모바일에서 만든 일정을 LB로 가져오기 위함
- **결론**: **Plan만 / 양방향 / 전용 캘린더 사용**

- 대상 앱: Life Binder Web (`D:\11_WorkSpace\Schedule_Binder`)
- 구성: 정적 웹앱 (`index.html` + `app.js` + `styles.css`)
- 배포: GitHub Pages (`.github/workflows/pages.yml`)
- 현재 데이터 저장: 브라우저 `localStorage` + Supabase 원격 저장
- 시간대: Asia/Seoul
- 일정 데이터 구조 특징: 한 일정 객체가 "계획(Plan)"과 "실행(Do)" 두 시간을 동시에 가짐
  - 계획: `start`, `end`, `categoryId`
  - 실행: `actualStart`, `actualEnd`, `actualCategoryId`, `actualText`

## 1. 연동 범위

### 1-1. 동기화 대상 ✅

- ✅ **(A) Plan만 동기화** — 실행(Do) 기록은 LB 안에만 머무름

### 1-2. 동기화 방향 ✅

- ✅ **(c) 양방향**
  - LB → GCal: LB에서 계획을 만들거나 수정하면 GCal 전용 캘린더에 반영
  - GCal → LB: 모바일 GCal에서 전용 캘린더에 추가/수정한 일정을 LB의 Plan으로 가져옴

### 1-3. 동기화 트리거 ✅ (양방향 사용 시나리오에 맞춰 자동 결정)

- ✅ **LB → GCal**: 일정 저장 시 즉시 + 앱 시작 시 1회 보정 동기화
- ✅ **GCal → LB**: 앱 시작 시 1회 + 일정 간격 폴링(아래에서 주기 선택) + 수동 "지금 동기화" 버튼
- 정적 사이트라 GCal 푸시 알림(웹훅)을 받을 수 없어서 **폴링이 필수**입니다.

**GCal 폴링 주기 선택**: ✅ **5분 (기본값)** — 설정 탭에서 1분/5분/15분/수동 변경 가능 *(2026-06-10 추천 채택)*

### 1-4. 동기화 대상 캘린더 ✅ (양방향 + Plan만 조합에서는 사실상 강제)

- ✅ **(i) 새 캘린더 "Life Binder"를 앱이 자동 생성** 후 거기에만 동기화
  - 이유: 양방향 + Plan-only 조합에서는 GCal → LB 방향이 "전용 캘린더 안의 모든 이벤트를 LB Plan으로 가져온다"는 단순한 규칙이 됩니다. Primary 캘린더에 섞으면 회사 일정·생일·자동 추가된 이벤트까지 다 LB로 들어와서 노이즈가 큽니다.
  - 모바일 GCal에서는 좌측 메뉴에서 "Life Binder" 캘린더의 색·표시 여부를 자유롭게 켜고 끌 수 있습니다.

## 2. 인증

### 2-1. 인증 방식 (수정 불필요)

- 정적 사이트(GitHub Pages)이므로 **Google Identity Services (GIS) 클라이언트 사이드 OAuth 2.0** 사용
- 토큰은 `localStorage`에 저장, 만료 시 자동 재발급
- 스코프: `https://www.googleapis.com/auth/calendar` (전용 캘린더 자동 생성 권한 포함)

### 2-2. OAuth Client ID — **입력 필요**

사용자가 본인의 Google Cloud Console에서 발급한 OAuth 2.0 클라이언트 ID가 필요합니다.

- [ ] 이미 발급받았음 → Client ID: (해당 없음)
- [x] 아직 발급 안 함 → 작업과 함께 **README에 발급 절차 안내 문서**도 추가 요청

**내 선택**: ✅ **아직 발급 안 함** — README에 발급 절차 추가 완료. 발급 후 앱 설정 탭에 입력하면 됨.

### 2-3. Client ID 입력 위치 — **입력 필요**

- [ ] (가) `app.js` 상단 상수에 하드코딩 — 간단하지만 git에 노출됨
- [x] (나) 앱 안에 "설정" UI에서 사용자가 입력, `localStorage`에 저장 **(권장)**
- [ ] (다) `.env`-like 별도 파일 (예: `config.local.js`) — `.gitignore`로 제외

**내 선택**: ✅ **(나)** — 설정 탭 입력, `localStorage`(`life-binder-gcal-v1`) 저장 *(권장 채택)*

## 3. 이벤트 매핑 규칙

### 3-1. 시간 매핑 (수정 불필요)

- LB 일정 `start` / `end` → GCal `start.dateTime` / `end.dateTime`
- 시간대: `Asia/Seoul`

### 3-2. 매핑 추적 방식 ✅

- LB 블록 `id`를 GCal 이벤트 `extendedProperties.private.lifeBinderId`에 저장
- 매핑 테이블을 `state.gcalSync = { [blockId]: { eventId, etag, updated } }` 형태로 localStorage에 보존
- GCal에서 새로 만든 이벤트(= `lifeBinderId` 없음)는 LB에 새 Plan 블록으로 추가되고, 이때 LB 측에서 새 `id`를 발급해 GCal 이벤트에 거꾸로 써넣어 다음부터는 매핑됨

### 3-3. 카테고리 ↔ GCal 색상 (colorId) — **확인 필요**

GCal 이벤트 색상은 11개 고정. 추천 매핑은 아래와 같습니다.

| LB 카테고리 | LB 색상 (참고) | 추천 colorId | 변경 시 입력 |
|---|---|---|---|
| 주업무 | 노랑 `#f2c94c` | **5 (Banana)** | `[ 입력: 1~11 ]` |
| 보조업무 | 핑크 `#e85d8f` | **4 (Flamingo)** | `[ 입력: 1~11 ]` |
| 신앙/가정/봉사 | 초록 `#2ec27e` | **10 (Basil)** | `[ 입력: 1~11 ]` |
| 자기개발 | 파랑 `#4dabf7` | **9 (Blueberry)** | `[ 입력: 1~11 ]` |
| 휴먼 네트워크 | 주황 `#ff922b` | **6 (Tangerine)** | `[ 입력: 1~11 ]` |

**내 선택**: ✅ **추천 채택** (주업무 5 / 보조업무 4 / 신앙·가정·봉사 10 / 자기개발 9 / 휴먼 네트워크 6)

### 3-4. GCal → LB로 가져올 때 기본 카테고리 — **입력 필요** (양방향 때문에 새로 생긴 결정)

모바일 GCal에서 새 일정을 만들면 카테고리 정보가 없습니다. LB로 가져올 때 어느 카테고리로 분류할지 정해야 합니다.

- [ ] 주업무
- [ ] 보조업무
- [ ] 신앙/가정/봉사
- [ ] 자기개발
- [ ] 휴먼 네트워크
- [x] **GCal 이벤트 색상(colorId)을 보고 §3-3 매핑의 역방향으로 자동 분류, 색상이 기본값이면 "주업무"로 (권장)**

**내 선택**: ✅ **색상 역방향 자동 분류, 기본 "주업무"** *(권장 채택)*

### 3-5. 이벤트 본문 매핑 (수정 불필요)

- GCal `summary` (제목) ↔ LB 일정 제목
- GCal `description` ↔ LB 일정 `memoText`
- 첨부 사진(`photos`)은 동기화 제외 (LB 안에서만)

## 5. 충돌 / 에러 처리

### 5-1. 충돌 처리 — **확인 필요** (양방향이라 중요)

같은 일정을 LB와 GCal에서 거의 동시에 수정한 경우의 규칙.

- [x] (1) **양쪽 `updated` 타임스탬프 비교 → 더 최신이 이김 (권장)**
- [ ] (2) LB가 항상 우선 — "LB가 마스터, GCal은 거울"
- [ ] (3) GCal이 항상 우선
- [ ] (4) 충돌 시 LB 안에 알림 띄우고 사용자가 선택

**내 선택**: ✅ **(1) 최신 우선** *(권장 채택)* — LB 쪽은 `block.planUpdatedAt`, GCal 쪽은 `event.updated`로 비교

### 5-2. 삭제 처리 — **확인 필요** (양방향이라 중요)

- 한쪽에서 일정을 삭제했을 때 다른 쪽도 자동 삭제할지 여부.
- [x] (가) **양쪽 모두 즉시 삭제 (권장)**
- [ ] (나) 삭제는 절대 자동 전파하지 않음 (수동 정리)
- [ ] (다) LB에서 삭제 → GCal도 삭제. 그러나 GCal에서 삭제 → LB는 그대로 둠 (모바일 실수 방지)

**내 선택**: ✅ **(가) 양쪽 모두 즉시 삭제** *(권장 채택)* — LB에서 계획을 "취소" 처리한 경우에도 GCal에서는 제거됨

### 5-3. 오프라인/실패 처리

- 실패한 동기화는 `state.gcalSyncQueue`에 쌓아두고 다음 성공 시 일괄 처리
- 동일 동기화가 **N회** 이상 연속 실패하면 사용자에게 알림

**N회**: ✅ **3회** *(추천 채택)* — 3회 연속 실패 시 알림. 동기화는 diff 스캔 방식이라 실패분은 다음 성공 시 자동 반영됨(별도 큐 불필요)
